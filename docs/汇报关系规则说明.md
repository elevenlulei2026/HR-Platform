# 汇报关系规则说明

本文约定：**默认管理汇报线（完整汇报链）如何由组织主数据推导**。  
实现衍生计算、花名册展示、审批链预填、与 `reporting_line` 手工维护对齐时，以本文为准。

相关字段（见 `docs/领域模型与表设计-MVP.md`）：

| 业务称呼 | 主数据字段 | 说明 |
| --- | --- | --- |
| 部门负责人 / 组织负责人 | `organization.org_leader_no` | 本部门负责人 |
| 分管领导 | `organization.supervising_leader_no` | 可为空 |
| 上级部门 | `organization.parent_id` / `parent_code` | 组织树向上追溯 |

员工所属部门取自**当前有效主任职**对应组织（`employee_assignment`，`is_primary=1` 且在查询日生效）。组织树、负责人、分管领导均按查询日 `asOfDate` 取版本快照。

---

## 1. 核心规则（一句话）

> 从员工本人出发，按所属部门 → 上级部门 → … → 根部门的顺序，**每一层先挂「部门负责人」，再挂「分管领导」（若有）**，依次拼成完整汇报线；某一角色未配置则跳过该角色。

---

## 2. 推导算法

设：

- 员工为 \(E\)
- 所属部门为 \(O_1\)，其上级链为 \(O_1 \rightarrow O_2 \rightarrow \cdots \rightarrow O_n\)（\(O_n\) 为根或不再向上）
- 部门 \(O_i\) 的负责人为 \(L_i\)，分管领导为 \(S_i\)（均可空）

则完整汇报线节点序列为：

```text
[E] + flatten( for i = 1..n:  [L_i?, S_i?] )
```

伪代码：

```text
chain = [员工]
org = 员工所属部门（asOfDate）
while org != null:
  if org.org_leader 存在且尚未入链:  append org.org_leader
  if org.supervising_leader 存在且尚未入链:  append org.supervising_leader
  org = org.parent（asOfDate）
```

展示时用 `>` 连接工号或姓名，例如：`A > B > C > D > E > F`。

### 2.1 本层汇报（部门内）

仅看所属部门一层：

| 条件 | 本层汇报关系 |
| --- | --- |
| 有负责人、有分管领导 | 员工 → 负责人 → 分管领导 |
| 仅有负责人 | 员工 → 负责人 |
| 仅有分管领导 | 员工 → 分管领导 |
| 两者皆空 | 本层无上级，继续向上层组织寻找 |

### 2.2 完整汇报线（跨上级部门）

对本部门及所有祖先部门，**重复「负责人 → 分管领导」拼接**，直到组织树顶端。

---

## 3. 完整示例

组织与角色配置：

```text
部门3
  负责人：E
  分管领导：F
  └── 部门2
        负责人：D
        分管领导：（无）
        └── 部门1
              负责人：B
              分管领导：C
              └── 员工 A（所属部门 = 部门1）
```

逐步推导：

| 步骤 | 部门 | 追加节点 | 当前链 |
| --- | --- | --- | --- |
| 0 | — | 员工本人 A | `A` |
| 1 | 部门1 | 负责人 B，分管领导 C | `A > B > C` |
| 2 | 部门2 | 负责人 D（无分管） | `A > B > C > D` |
| 3 | 部门3 | 负责人 E，分管领导 F | `A > B > C > D > E > F` |

**结论**：

- **本层汇报关系**：`A > B > C`
- **完整汇报线**：`A > B > C > D > E > F`

## 4. 补充规则（实现须遵守）

| # | 规则 | 说明 |
| --- | --- | --- |
| R1 | 空角色跳过 | 某部门未维护负责人或分管领导时，不写入该位，不留空位 |
| R2 | 自下而上 | 必须从员工所属部门向根遍历，禁止先拼上层再拼本层 |
| R3 | 层内顺序固定 | 同一部门内顺序恒为：**负责人 → 分管领导**，不可对调 |
| R4 | 去重保留先出现 | 同一人在多层重复出现时，**只保留第一次**（更靠近员工的一侧）；避免环与重复展示 |
| R5 | 员工即本部门负责人 | 若员工本人 = 本部门 `org_leader`，本人不作为上级入链；本层从分管领导起算（若有），否则直接进入上级部门 |
| R6 | 员工即本部门分管领导 | 若员工本人 = 本部门 `supervising_leader`，跳过该角色，进入上级部门继续拼接 |
| R7 | 主任职为准 | 兼岗不改变默认汇报线；衍生链始终基于主任职所属组织 |
| R8 | 生效日期 | 任职、组织负责人/分管领导、组织上下级变更均按 `asOfDate` 快照计算；变更须带 `effective_start_date` / `effective_end_date` |
| R9 | 停用组织 | 历史上有效的停用组织在历史 `asOfDate` 下仍参与推导；当前日仅使用查询日有效的组织版本 |

## 5. 与 `reporting_line` 表的关系

| 概念 | 来源 | 用途 |
| --- | --- | --- |
| **默认完整汇报线** | 本文算法，由组织树 + 负责人/分管领导衍生 | 展示、审批预填、组织洞察等只读能力 |
| **`reporting_line`（DIRECT / DOTTED）** | 页面或接口维护的成对上下级 | 直接上级覆盖、虚线汇报等需显式落库的场景 |

约定：

1. **默认链**不强制物化为 `reporting_line` 多行；可按需「按组织同步」生成 DIRECT 边。
2. 若存在有效的 **DIRECT** 手工记录，**直接上级**以手工为准；完整链仍可按本文继续向上补齐（从直接上级所属部门或手工链终点再走组织衍生，产品实现时二选一并在代码注释标明）。
3. **DOTTED** 不参与默认完整汇报线的主链拼接，仅作虚线关系单独展示。

> 实现：`/admin/employees/reporting-lines` 支持「按组织同步」将衍生链的首位上级落库为 DIRECT；列表「汇报线」列按本文算法衍生展示。DOTTED 不参与同步。

## 6. 验收用例（建议）

| 用例 | 给定 | 期望完整汇报线 |
| --- | --- | --- |
| 标准三级 | 本文 §3 示例 | `A > B > C > D > E > F` |
| 缺分管 | 部门2 无分管 | 链中不出现空位，直接 `… > C > D > E > …` |
| 缺负责人 | 某层仅有分管 S | 该层只追加 S |
| 重复人 | 部门1 分管 = 部门2 负责人 = X | X 只出现一次 |
| 本人是负责人 | A = 部门1 负责人，分管为 C | `A > C > …`（不出现第二个 A） |
| 历史快照 | 改负责人后按旧 `asOfDate` 查 | 仍见变更前链 |
