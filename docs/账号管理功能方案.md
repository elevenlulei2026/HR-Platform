# 账号管理功能方案

> **用途**：定义平台登录账号（`sys_user`）的管理能力、与员工档案「AD账号」的对应关系、角色绑定与生命周期规则。  
> **何时读**：实现账号管理页、员工开号、AD/登录名同步、离职禁用账号前，**先读本文**。  
> **状态**：方案已优化定稿，待开发（尚未实现管理端页面）。  
> **相关文档**：`docs/领域模型与表设计-MVP.md` §2 / §4.2、`docs/员工档案字段要求.md`（联系与账号）、`docs/生效日期功能规范.md`、`docs/MVP-AI开发路线图.md` Slice 1 / 3、`AGENTS.md`。

## 1. 背景与目标

### 1.1 现状

| 能力 | 现状 |
| --- | --- |
| 登录鉴权 | 已有：`POST /api/v1/auth/login`、`sys_user`、JWT |
| 种子账号 | Flyway 写入 `admin` / `manager` / `hr` / `employee`（密码均为演示用） |
| 角色绑定 API | 已有：`GET/PUT /api/v1/users/{id}/roles`（当前权限为 `permission:manage`） |
| RBAC 管理页 | 已有：`/admin/platform/permissions`（角色 / 菜单 / 权限点） |
| 账号管理页 | **无**：无法在 UI 中创建、启停、绑员工、重置密码 |
| 员工 AD 账号 | 档案主档字段 `ad_account`，目前为手填文本，与登录未强制一致 |
| 双向外键 | 模型已有：`sys_user.employee_id`、`employee.user_id`，管理闭环未落地 |
| 请求鉴权 | `AuthFilter` 验 JWT 后加载权限，**尚未**每次校验 `sys_user.status` |

### 1.2 目标

1. 在 Admin 端提供完整的**账号管理**（列表、创建、启停、重置密码、角色绑定）。
2. **员工档案「AD账号」即本系统登录账号**（`sys_user.username`）；二者在绑定后必须一致。
3. 账号可与员工一对一绑定，可与多个角色绑定（复用 `user_role`）。
4. 支持无员工的**系统账号**（如 `admin`）。
5. 写操作鉴权 + 审计；离职等生命周期可联动停用账号。
6. 停用后**立即**不可访问受保护接口（不依赖 JWT 自然过期）。

### 1.3 非目标（MVP 不做）

- SSO / AD / LDAP 目录同步（字段语义预留「AD」，登录仍为本地用户名密码）。
- MFA、完整密码策略引擎、登录失败锁定（`locked_until` 可预留列，逻辑后补）。
- 字段级权限、临时授权、委托登录。
- 在账号管理中直接编辑员工主数据（开号以外的人事字段仍走档案）。
- 暴露或维护 `manager_user_id`（历史工作流演示字段；审批应走组织负责人 / 汇报线）。
- 账号物理删除、随意解绑、按组织裁剪账号列表数据范围。
- 待入职（`CANDIDATE`）自动/手工开号；入职完成自动开号（后补开关）。
- 改登录名后主动踢掉全部旧会话（MVP 以 status 校验保证停用生效即可；改名后旧 token 仍可凭 `userId` 访问至过期，可接受）。

## 2. 核心概念

### 2.1 概念关系

```text
员工档案.AD账号 (ad_account)  ════════  sys_user.username（登录名）
         │                                    │
         └────── employee.user_id ────────────┘
                       ↕
                 sys_user.employee_id

sys_user ── N:M ── role ── N:M ── permission
                     └── data_scope / org_scope
```

| 概念 | 说明 |
| --- | --- |
| 登录账号 | `sys_user` 一条记录，用于登录与鉴权 |
| 登录名 | `sys_user.username` |
| AD账号 | 员工档案字段 `ad_account`；**员工账号场景下与登录名同义** |
| 员工绑定 | `sys_user.employee_id` ↔ `employee.user_id`，一人最多一账号 |
| 角色绑定 | `user_role`，一账号可多角色；数据范围跟角色走 |
| 系统账号 | 无员工绑定的账号（`employee_id` 为空），登录名独立，不依赖 AD |

**原则**：账号管「谁能登录、有何系统权限」；员工管「人事身份」。员工侧登录身份通过 **AD账号** 表达，不另建第二套用户名。

### 2.2 两类账号

| 类型 | `accountType`（派生） | 登录名来源 | 员工绑定 |
| --- | --- | --- | --- |
| 员工账号 | `EMPLOYEE` | **必须等于**当前有效档案 AD账号 | 必填，一对一 |
| 系统账号 | `SYSTEM` | 手工指定 `username` | 无 |

- `accountType` **不落库**，由 `employee_id IS NULL` 推导：`NULL` → `SYSTEM`，非空 → `EMPLOYEE`。
- 工号（`employee_no`）仅用于搜索、展示与业务主键，**不是**默认登录名。

### 2.3 模块分层（必须遵守）

依赖方向：`platform ← core ← modules`。**禁止** `platform` 依赖 `core` / `modules`。

| 职责 | 归属 | 说明 |
| --- | --- | --- |
| 账号本体 | `platform.auth.UserAccountService` | 系统账号 CRUD、启停、重置密码、本人改密、`must_change_password`、按 `employeeId` 停用 |
| 员工开号 / 绑定 / 改 AD 同步登录名 | `core.employee.EmployeeAccountBindingService`（名称可微调） | 校验员工与 AD、写双向外键、同步主档 AD；**内部调用** platform 创建/改名账号 |
| 档案侧改 AD（已开号） | 同上绑定服务 | 档案 `PUT` / 批导不得各自改一半 |
| 离职停用 | `modules` 发领域事件 → platform `disableByEmployeeId` | 禁止离职模块直接改 `sys_user` |
| 列表/详情组装 | platform Controller 可调 core **只读**查询员工展示字段；或列表 DTO 由应用层组装 | 读可跨层，写必须走唯一入口 |

```text
Admin API
  ├─ 系统账号 / 启停 / 重置密码 / 改密     → platform.UserAccountService
  ├─ 员工开号 / 改 AD·登录名 / 档案快捷开号 → core.EmployeeAccountBindingService
  │                                            └─ calls → UserAccountService
  └─ 离职生效事件                             → UserAccountService.disableByEmployeeId
```

**唯一写入口**：双向外键与「已开号 AD ↔ username」同步，只允许经 `EmployeeAccountBindingService`（或等价唯一编排）。档案普通编辑、账号管理页、批导/导入凡触及已绑定 AD，必须调用同一方法；否则拒绝或整单失败。

- `uk_sys_user_username`（已有）
- `uk_sys_user_employee_id`（`employee_id`，NULL 可多行系统账号）
- `idx_sys_user_status`（列表筛选）

**不落库字段**：`accountType`（派生）、列表用的 `employeeNo` / `employeeName` / `adAccount`（查询组装）。

### 3.2 `employee` / 主档版本（核心域）

| 字段 | 说明 |
| --- | --- |
| `employee.ad_account` | 投影/兼容字段；有值时业务唯一；长度 **VARCHAR(64)**（与 `username` 对齐；实现时 Flyway 将现有 128 收束为 64，超长数据迁移前校验） |
| `employee.user_id` | 绑定的 `sys_user.id`；**UNIQUE** |
| `employee_master_version.ad_account` | 版本化字段；**登录与开号只认当前有效版本** |

绑定后强制：

```text
当前有效 master.ad_account
  == employee.ad_account（投影保持一致）
  == 绑定 sys_user.username
```

> 领域模型原文见 `docs/领域模型与表设计-MVP.md` §4.2。实现本方案时同步修订该处说明。

### 3.3 规范化与唯一性

**规范化（写入前统一）**：

- `trim` 首尾空白；禁止空串（视为未填）。
- MVP **大小写敏感**（与现库 UNIQUE 行为一致）；文档与 UI 提示勿随意改大小写。
- 长度 ≤ 64。

| 约束 | 规则 |
| --- | --- |
| 登录名全局唯一 | `sys_user.username` |
| 一员工一账号 | `sys_user.employee_id` UNIQUE；`employee.user_id` UNIQUE |
| AD 有值时唯一 | 应用层强校验：相对其他员工**当前有效** AD + 全部 `sys_user.username` |
| 角色 | 标准 `user_role(user_id, role_id)` |

### 3.4 禁止物理删除

账号生命周期仅 `ACTIVE` ↔ `DISABLED`。**禁止**物理删除 `sys_user`，避免审批历史、审计、外键引用断链。

## 4. AD账号 ↔ 登录名：同步与校验规则

### 4.1 单一真相

- **鉴权真相**：`sys_user.username`（登录只认此字段）。
- **人事可见真相**：档案当前有效版本的「AD账号」。
- **绑定后强制**：当前有效 AD == 绑定账号的 `username`；`employee` 投影字段同步。
- **双向外键**：`employee.user_id` 与 `sys_user.employee_id` 必须成对一致，禁止只改一侧。

### 4.2 开号前置（员工账号）

1. 员工存在，状态为 `PROBATION` 或 `ACTIVE`（`TERMINATED` / `CANDIDATE` **禁止**新开；待入职开号为后补能力，见 §1.3）。
2. **当前有效 AD账号非空**；为空则阻断并提示先维护档案。
3. 该 AD 不与任何已有 `sys_user.username` 冲突，且不与其他员工当前有效 AD 冲突。
4. 该员工尚未绑定账号（`user_id` 为空 / 无对应 `sys_user.employee_id`）。

开号成功时（同一事务，经 `EmployeeAccountBindingService`）：

1. 调用 platform 创建 `sys_user`：`username = 规范化后的当前 AD`，`employee_id = employee.id`，`status = ACTIVE`。
2. `UPDATE employee.user_id = sys_user.id`。
3. 写入 `user_role`（可选角色列表）。
4. 设置初始密码哈希；默认 `must_change_password = true`。
5. 写审计日志。

### 4.3 档案侧变更 AD账号

| 场景 | 规则 |
| --- | --- |
| 未开号 | 允许修改；提交前做全局唯一预检（其他员工当前 AD + 全部登录名） |
| 已开号 | **改 AD = 改登录名**：走绑定服务；MVP **仅允许修正当前有效版本**（`editMode = CURRENT`），同事务更新：当前 master `ad_account`、`employee.ad_account`、`sys_user.username`；冲突则整单失败 |
| 已开号 + 未来版本 | **禁止**通过未来版本「预约改登录名」；未来版本若含不同 AD，开号/登录/唯一性校验**忽略**，只认当前有效版本 |
| 已开号且欲清空 AD | **禁止**；须先停用（MVP 不开放解绑） |
| 历史版本中的旧 AD | 保留作快照；登录只认当前绑定用户的 `username` |
| 批导 / 导入含 `adAccount` | 未开号：按普通档案规则；已开号：必须调绑定服务同步，或整行拒绝 |

权限：

- 未开号改 AD：员工编辑权限即可。
- 已开号同步改登录名：须同时具备员工编辑权限与 `user:manage`（MVP 不单拆 `user:rename`）。

### 4.4 账号管理页对登录名的处理

- 员工账号：列表/详情中登录名与 AD **只读展示**；变更走「变更 AD/登录名」专用动作（与档案相同领域方法）。
- 系统账号：创建时可指定 `username`；MVP **创建后不可改**。

### 4.5 停用 / 解绑 / 离职

| 动作 | 行为 |
| --- | --- |
| 停用 | `sys_user.status = DISABLED`；鉴权链路拒绝；档案 AD **保留**；绑定关系保留 |
| 启用 | 恢复 `ACTIVE`；若员工已 `TERMINATED` 则禁止启用 |
| 解绑 | **MVP 不做**（不开放清空双向外键、不改名释放 username） |
| 员工离职生效 | 领域事件 → `disableByEmployeeId`；AD 只读保留 |
| 复职 | **不**自动启用账号；管理员确认角色后手工启用 |

### 4.6 登录与会话规则

- 仅 `status = ACTIVE` 可签发 token。
- **每次受保护请求**（登录除外）须校验对应用户仍为 `ACTIVE`；非 ACTIVE 立即 401。实现上可在 `AuthFilter` / `RbacService.enrich` 查库（或极短 TTL 缓存）。
- 员工用户使用 **AD账号** 作为登录名（文案：「AD账号 / 登录名」）。
- `must_change_password = true` 时：
  - 登录成功可签发 token；
  - 前端拦截业务路由，引导改密；
  - 后端：除 `GET /auth/me`、`PUT /auth/password`、登出外，其余写接口与敏感读返回明确错误码（如 `PASSWORD_CHANGE_REQUIRED`），Slice C 落地。

## 5. 角色绑定

- 复用现有 `user_role` 与 `GET/PUT /api/v1/users/{id}/roles`。
- 仅可分配 `status = ACTIVE` 的角色。
- 允许 0 个角色；开号时可预选建议角色（如 `employee` / `hr`）。
- 数据范围不跟账号，跟角色。
- **防锁死**：禁止当前操作者移除自身最后一个具备 `user:manage` **或** `permission:manage` 的授权路径（含禁用自己、去掉关键角色）。判定须同时覆盖两条权限。

账号管理与「RBAC 权限」页分工：

| 页面 | 职责 |
| --- | --- |
| `/admin/platform/permissions` | 角色、菜单、权限点、角色数据范围 |
| `/admin/platform/users` | 账号、AD/员工绑定、账号↔角色挂接、启停、重置密码 |

**权限迁移（相对现状）**：

| 接口 | 现状 | 本方案 |
| --- | --- | --- |
| 角色/菜单/权限点 CRUD | `permission:manage` | 不变 |
| `GET/PUT /users/{id}/roles` | `permission:manage` | 改为 **`user:manage`**（账号管理职责） |
| 账号列表/创建/启停/重置/开号/改名 | — | `user:manage` |

实现时同步改 `RbacController` 中 roles 接口的鉴权注解与前端按钮权限；种子：`admin` 角色同时拥有二者。

## 6. API 契约（设计稿）

实现时先写入 `shared/api.interface.ts`，再写 Java / 前端。路径前缀：`/api/v1/`。JSON camelCase。

### 6.1 类型（摘要）

```ts
export type UserAccountType = "EMPLOYEE" | "SYSTEM"; // 派生，不落库

export type SysUserAccount = {
  id: string;
  username: string;
  accountType: UserAccountType;
  /** 系统账号用 sys_user.display_name；员工账号为当前员工姓名（join，非库字段） */
  displayName?: string;
  status: UserStatus; // ACTIVE | DISABLED
  employeeId?: string;
  employeeNo?: string;
  employeeName?: string;
  adAccount?: string; // 员工账号时与 username 同值
  roles: string[];
  mustChangePassword: boolean;
  lastLoginAt?: string;
  createdAt?: string;
  updatedAt?: string;
};

export type SysUserListQuery = {
  keyword?: string; // 登录名 / 展示名 / 工号 / 姓名 / AD
  status?: UserStatus | "ALL";
  roleCode?: string;
  accountType?: UserAccountType | "ALL";
  boundEmployee?: "YES" | "NO" | "ALL";
  page: number;
  pageSize: number;
};

/** 正式开号入口：仅系统账号走本接口；员工账号见 open-account */
export type SysUserCreateRequest = {
  accountType: "SYSTEM";
  username: string;
  displayName?: string;
  password: string;
  roleCodes?: string[];
  mustChangePassword?: boolean; // 默认 true
};

export type SysUserUpdateRequest = {
  displayName?: string; // 仅 SYSTEM
  status?: UserStatus;
  // 不含 roleCodes：角色只走 PUT /users/{id}/roles
};

export type OpenEmployeeAccountRequest = {
  password: string;
  roleCodes?: string[];
  mustChangePassword?: boolean; // 默认 true
};

export type ResetPasswordRequest = {
  newPassword: string;
  mustChangePassword?: boolean; // 默认 true
};

export type RenameLoginRequest = {
  newAdAccount: string; // 新 AD = 新 username；仅 EMPLOYEE
};

export type ChangePasswordRequest = {
  oldPassword: string;
  newPassword: string;
};
```

### 6.2 接口一览

| 方法 | 路径 | 权限 | 说明 |
| --- | --- | --- | --- |
| GET | `/users` | `user:manage` | 分页列表 |
| GET | `/users/{id}` | `user:manage` | 详情 |
| POST | `/users` | `user:manage` | **仅创建系统账号**（`accountType: SYSTEM`） |
| PUT | `/users/{id}` | `user:manage` | 更新展示名（SYSTEM）/ 状态；**不含角色** |
| POST | `/users/{id}/reset-password` | `user:manage` | 管理员重置密码 |
| POST | `/users/{id}/rename-login` | `user:manage` | 变更 AD/登录名（员工账号；转调绑定服务） |
| GET/PUT | `/users/{id}/roles` | `user:manage` | 角色全量查询/覆盖（权限从 `permission:manage` 迁出） |
| POST | `/employees/{id}/open-account` | `user:manage` | **员工开号正式入口**（`username = 当前 AD`） |
| PUT | `/auth/password` | 已登录本人 | 修改自己的密码 |

响应统一：`{ code, message, data, traceId }`；分页：`{ items, total, page, pageSize }`。

**入口收敛**：

- 员工开号：只认 `POST /employees/{id}/open-account`（账号页「新建员工账号」也调此接口）。
- 角色变更：只认 `PUT /users/{id}/roles`。
- 改 AD/登录名：账号页与档案页均调同一绑定服务（可共用 `rename-login` 或档案更新内嵌调用）。

### 6.3 包与依赖

| 组件 | 包 |
| --- | --- |
| 账号本体服务 | `com.hrplatform.platform.auth.UserAccountService` |
| 员工绑定/开号/改名 | `com.hrplatform.core.employee.EmployeeAccountBindingService` |
| 账号 Controller | `com.hrplatform.platform.web.user`（列表/系统账号/启停/重置/roles） |
| 开号 Controller | `com.hrplatform.core.web.employee`（`open-account`；rename 可挂此处或 platform 转调 core） |

密码：

- 新账号 / 新哈希优先 BCrypt；校验兼容现有 `sha256:` 前缀种子账号。
- MVP 最低规则：长度 ≥ 8；不得与 username（忽略大小写）相同；明文仅经 HTTPS 传输；审计只记「已重置/已改密」，不记明文。

### 7.2 账号管理页

页面：`AdminUsersPage`（对齐现有 Admin 列表模式）。

- 危险操作：`ConfirmDialog`（停用、重置密码、变更登录名）。

- 反馈：`sonner`；图标仅 `lucide-react`。

新建分流：

1. **员工账号**：选员工 → 带出 AD、姓名 → AD 为空则阻断 → 登录名只读 = AD → 选角色 + 初始密码 → 调用 `POST /employees/{id}/open-account`。
2. **系统账号**：手工 username、displayName、角色、密码 → 调用 `POST /users`，角色可创建后立即 `PUT .../roles`。

### 7.3 员工档案联动

- 字段文案：AD账号旁注「即系统登录账号」。
- 展示开通状态：未开通 / 已开通（ACTIVE）/ 已停用。
- 具备 `user:manage` 时提供：「开通登录账号」「重置密码」「停用账号」。
- 已开号编辑 AD：走同步改登录名规则（见 §4.3）；无 `user:manage` 时字段只读或提交被拒。

字段要求文档（实现阶段）改为：

> AD账号：系统登录账号（最长 64）；开通登录后与平台用户名一致。未开通时可手填；已开通时修改将同步登录名（需员工编辑 + `user:manage`）。

### 7.4 登录与顶栏

- 登录页提示：员工使用 AD账号 登录；可保留演示种子说明。
- 顶栏用户区：员工账号显示**员工姓名**（join）；系统账号显示 `displayName`，否则 `username`；提供「修改密码」（本人）。
- `must_change_password`：登录后强制进入改密流程，未完成前不可进入业务页。

## 8. 权限与安全

### 8.1 权限点

| code | 说明 |
| --- | --- |
| `user:manage` | 账号列表/创建系统账号/启停/重置密码/角色挂接/开号/变更登录名 |
| `permission:manage` | 角色、菜单、权限点、数据范围（不含账号运维） |

种子：二者均赋给 `admin` 角色；账号管理菜单绑定 `user:manage`。

账号列表为平台运维能力，MVP **不做**按组织数据范围裁剪（避免跨部门账号无法维护）。持 `user:manage` 即全局可见。

### 8.2 安全要求

- 所有写接口后端鉴权；禁止仅前端藏按钮。
- 响应与审计详情**禁止**出现明文密码或 `passwordHash`。
- 开号、改 AD/登录名、改角色、启停、重置密码 → 写 `audit_log`。
- 防自我锁死（§5）。
- 敏感操作确认弹窗。
- 受保护请求校验 `ACTIVE`（§4.6）。
- **禁止**物理删除账号。

## 9. 生命周期与事件

```text
创建 ACTIVE ──停用 / 离职事件──► DISABLED ──启用（员工未离职）──► ACTIVE

登录签发：仅 ACTIVE
请求鉴权：每次校验 ACTIVE
must_change_password=true → 强制改密后再用业务功能
禁止物理删除
```

| 业务事件 | 账号动作 |
| --- | --- |
| 入职完成 | MVP **手工开号**；后补「自动开号」开关（与待入职开号一并后补） |
| 离职生效 | 自动停用绑定账号 |
| 复职 | 不自动启用；管理员手工启用并核对角色 |

跨模块：离职用领域事件通知平台停用账号；统一调用 `UserAccountService.disableByEmployeeId`。

## 10. 垂直切片交付计划

禁止一次铺开；建议三刀：

### Slice A — 账号底座

- Flyway：`sys_user` 增列、`employee_id` / `user_id` 唯一、`ad_account` 长度对齐 64、权限点 `user:manage` 与菜单种子。
- 契约 + `UserAccountService` + 列表 / 系统账号创建与更新 + 重置密码 + 启停。
- 鉴权链路增加 `ACTIVE` 校验（停用立即失效）。
- 页面：`/admin/platform/users`（系统账号主路径先通）。
- 验收：可创建系统账号并登录；停用后已有 token 立即不可用；审计有记录。

### Slice B — AD = 登录名 + 员工绑定（核心）

- `EmployeeAccountBindingService`：按员工开号、双向外键、角色绑定。
- `POST /employees/{id}/open-account`；档案 AD 唯一预检；档案「开通账号」与状态展示。
- `PUT /users/{id}/roles` 权限迁至 `user:manage`。
- 验收：无 AD 无法开号；一人不能两账号；开号后可用 AD 登录；角色变更后 `/auth/me` 权限更新。
- 联调：演示账号 `manager` / `hr` / `employee` 绑定真实员工或停用，避免与真实 AD 冲突。

### Slice C — 安全与闭环

- 已开号改 AD 同步登录名（当前版本 CURRENT）；本人改密；`must_change_password` 强制（前端 + 后端错误码）。
- 离职自动停用；防自我锁死（同时覆盖 `user:manage` / `permission:manage`）。
- 验收：改 AD 后旧登录名无法登录、新名可登录；离职后 Disabled；重置后首次须改密。

## 11. 验收标准（总览）

1. Admin 可在 `/admin/platform/users` 管理账号全生命周期（创建系统账号、启停、重置密码、角色）。
2. 员工账号登录名与档案当前有效 AD账号一致；开号以 AD 为准，不以工号为准。
3. 员工与账号一对一绑定；系统账号可不绑员工；`accountType` 由绑定状态派生。
4. 账号可绑定多个角色，权限即时生效；角色接口权限为 `user:manage`。
5. 写操作需 `user:manage`（本人改密除外）+ 审计；密码不明文落库/回传；禁止物理删除。
6. 停用与离职联动后，**已有 JWT 立即失效**（status 校验）。
7. platform / core 分层不被破坏：员工绑定写路径在 core，账号本体在 platform。

## 12. 定稿决策一览

| 决策项 | 定稿 |
| --- | --- |
| 员工登录名 | **AD账号**（当前有效 `ad_account` ≡ `username`） |
| 工号 | 仅搜索/展示，不作默认 username |
| 无 AD | 不能开员工登录账号 |
| 已开号改 AD | 同步改 username；仅修正当前有效版本；同事务 |
| 未来版本改 AD | MVP 禁止用于改登录名 |
| 系统账号 | 允许无员工、无 AD（如 admin） |
| 登录名创建后 | 系统账号 MVP 不可改；员工账号经「变更 AD/登录名」改 |
| `accountType` | 派生，不落库 |
| `display_name` | 仅系统账号；员工姓名运行时 join |
| 开号 API | `POST /employees/{id}/open-account` 为正式入口 |
| 角色 API | 仅 `/users/{id}/roles`；权限改为 `user:manage` |
| Update API | 不含 `roleCodes` |
| `manager_user_id` | 管理端不暴露，逐步废弃 |
| 入职自动开号 / 待入职开号 | MVP 不做，后补 |
| 离职 | 自动停用绑定账号 |
| 解绑 / 物理删除 | MVP 不做 / 禁止 |
| 会话 | 每次请求校验 ACTIVE；改名踢会话后补 |
| 密码哈希 | 新账号 BCrypt，旧 `sha256:` 兼容验证 |
| 模块分层 | 绑定在 core，账号本体在 platform |
