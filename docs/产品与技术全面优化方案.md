# HR Platform 产品与技术全面优化方案

> 对照成熟 SaaS HR（Workday、SAP SuccessFactors、北森、i人事、钉钉智能人事）与当前《HR系统完整系统方案》，从产品和研发双视角给出的优化结论与落地建议。

## 1. 现状评估

### 1.1 已有优势

| 维度 | 评价 |
| --- | --- |
| 业务全景 | 覆盖员工生命周期八大环节，模块边界清晰 |
| 架构分层 | 业务应用 / 协同流程 / 平台能力 / 数据基础四层合理 |
| 主数据意识 | Employee 与 Assignment 分离、异动历史、模块间通过 employee_id 关联 |
| 安全合规 | 功能/数据/字段权限、薪酬高敏、审计要求明确 |
| 实施节奏 | MVP 聚焦入转调离 + 底座，避免过早铺开 ATS/薪酬引擎 |
| 技术选型 | 模块化单体 + React/Spring Boot，与团队能力和交付效率匹配 |

### 1.2 与成熟 SaaS 的关键差距

| 差距项 | 成熟 SaaS 常见做法 | 当前方案 | 优化建议 |
| --- | --- | --- | --- |
| **生效日期（Effective Dating）** | 组织、任职、薪酬、合同均支持未来生效与历史追溯 | 有历史态概念，未作为一等公民 | 所有主数据变更必须带 `effective_date`，查询支持「截至某日」快照 |
| **三端体验** | 员工自助 ESS、主管 MSS、HR 后台分离 | 角色有定义，端未拆分 | MVP 仅 `admin/`；`ess/` `mss/` 后补，共享组件库与 API |
| **编制约束** | 招聘/入职/调岗实时校验 HC | 规划有编制，执行链路未闭环 | 入职、调岗、返聘前调用编制校验服务 |
| **流程可配置度** | 表单+流程+规则三元配置 | 有流程/表单中心，缺规则引擎边界 | 明确：流程管审批链，规则管计算与校验，表单管采集 |
| **指标口径** | 指标字典 + 维度 + 版本 | 报表指标列表化 | 建立 `MetricDefinition` 与统一数仓口径 |
| **集成可靠性** | 幂等键、死信队列、对账 | 原则有，缺实现规范 | 集成事件表 + external_id 映射 + 重试策略 |
| **多法人/多地区** | 法人实体、税务规则、日历分域 | 有 LegalEntity，缺地区规则包 | 规则按 `legal_entity_id + region` 作用域配置 |
| **租户隔离** | 若对外 SaaS 需 tenant_id 全链路 | 未涉及 | 单企业部署可暂缓；若 SaaS 化从 Day1 加 tenant_id |
| **批量与迁移** | 导入模板、校验报告、回滚 | 数据治理有原则 | MVP 即提供组织/员工 Excel 导入与错误行报告 |
| **AI 能力边界** | 简历解析、政策问答、异常检测 | 列为远期 | MVP 不做；预留简历文本字段与向量检索扩展点 |

## 2. 产品架构优化

### 2.1 用户端拆分（长期推荐，MVP 仅 Admin）

```text
ESS（员工自助）          MSS（主管自助）           Admin（HR/系统）
├─ 我的档案              ├─ 团队花名册             ├─ 组织岗位
├─ 我的待办              ├─ 团队待办               ├─ 员工管理
├─ 请假/加班（后期）      ├─ 审批（转正/离职等）     ├─ 入转调离
├─ 证明申请              ├─ 团队编制视图           ├─ 流程/权限配置
└─ 政策查询              └─ 团队报表摘要           └─ 审计/集成
```

**MVP 范围**：仅实现 **Admin** 列功能；ESS / MSS 列能力在 MVP 中由 HR 在 Admin 代操作（如代发起证明、入职办理、待办审批）。

**长期原则**：同一业务能力（如转正）在 ESS 发起、MSS 审批、Admin 监控，共用一套 API 与流程实例；后补 MSS/ESS 前端时无需改后端契约。

### 2.1.1 Admin 端 UI/UX 基准（已由 client/ 固化）

交互与信息架构以 **`client/`** 已实现页面为实践基准，规范见 **`docs/前端UI与信息架构规范.md`**。核心约定：

| 维度 | 约定 |
| --- | --- |
| 导航 | 顶栏一级模块 + Mega Menu 二级入口；**非左侧边栏** |
| 详情 | 列表页 + 右侧 Sheet 抽屉（查看/编辑/新建）；员工档案 6 Tab（个人信息/工作信息/员工服务/背景信息/人才发展/异动），覆盖 27 项二级模块（Slice 7） |
| 全局 | 命令面板 `Ctrl/Cmd+K`、面包屑、暗色模式、Toast 反馈 |
| 列表 | 筛选条 + 表格 sticky 表头 + 分页 + loading/error/empty 三态 |
| 权限 | 菜单项 `permission` 点位；无权限灰显；导出单独权限 + 审计 |
| 敏感数据 | 脱敏展示 + 标签；与加密存储、审计读一致 |
| 主数据 | 任职/组织/汇报展示 effective dating；支持 `asOfDate` |

正式 `client/` 统一使用 shadcn/ui + Tailwind；禁止原生 `input` / `select` / `textarea`。

### 2.2 MVP 范围微调（建议）

在原文档 MVP 基础上做以下调整：

**新增进 MVP（低成本、高价值）**

1. 生效日期与任职版本查询（「截至某日」组织/汇报树）
2. 员工/组织 Excel 导入导出（带校验报告）
3. 证明开具（模板 + 审批，SSC 高频场景）
4. 编制占用/释放与入职、调岗联动校验（可先简化为部门级 HC）

**明确延后（避免 MVP 膨胀）**

1. **MSS / ESS 前端**（主管自助、员工自助页面与 Layout；路由可预留）
2. 完整招聘 ATS（MVP 仅保留「手动创建待入职」入口）
3. 薪酬规则引擎（MVP 仅薪资档案只读占位）
4. 绩效校准会议（MVP 不做校准流）

### 2.3 核心业务流程状态机（MVP 必须固化）

| 流程 | 关键状态 | 终态副作用 |
| --- | --- | --- |
| 入职 | 草稿 → 待入职 → 入职中 → 已入职 / 已取消 | 创建 Employee + Assignment + ReportingLine |
| 转正 | 试用中 → 待转正 → 审批中 → 已转正 / 驳回 | 更新 Assignment、写入 Movement |
| 调岗调薪 | 草稿 → 审批中 → 待生效 → 已生效 | 关闭旧 Assignment、创建新 Assignment（带 effective_date） |
| 离职 | 申请中 → 审批中 → 交接中 → 待结算 → 已离职 | 任职结束、账号禁用、触发审计 |

状态机定义须落库为枚举 + 转换表，禁止在 Controller 硬编码分支。

## 3. 技术架构优化

### 3.1 模块化单体包结构（Java）

```text
com.hrplatform
├── platform/                 # 平台内核（禁止依赖业务模块）
│   ├── auth
│   ├── workflow
│   ├── notification
│   ├── audit
│   ├── file
│   └── config（字典/编码/自定义字段）
├── core/                     # 主数据内核
│   ├── organization
│   ├── employee
│   └── headcount
├── modules/                  # 业务模块（可依赖 platform + core）
│   ├── onboarding
│   ├── employee_relations
│   ├── recruitment        # 阶段3
│   ├── attendance
│   ├── payroll
│   ├── performance
│   ├── learning
│   └── offboarding
├── integration/              # 外部系统适配
└── shared-kernel/            # 公共类型、事件、异常
```

**依赖规则**：`modules/*` → `core` → `platform`；禁止 `core` 反向依赖 `modules`；跨模块通信用领域事件，禁止跨模块直接改表。

### 3.2 数据模型关键补充

#### 生效日期字段（所有主数据变更）

```sql
-- 示例：任职记录
effective_start_date DATE NOT NULL,
effective_end_date   DATE NULL,      -- NULL 表示当前有效
is_primary           TINYINT(1) DEFAULT 1,
status               VARCHAR(32) NOT NULL
```

#### 领域事件表（异步解耦）

```text
domain_event
├─ id, event_type, aggregate_type, aggregate_id
├─ payload_json, occurred_at, published_at
└─ status (pending/published/failed)
```

典型事件：`EmployeeHired`、`AssignmentChanged`、`EmployeeTerminated`。

### 3.3 API 设计规范

| 规范 | 说明 |
| --- | --- |
| 契约优先 | 先改 `shared/api.interface.ts`，再写 Controller/DTO |
| 版本 | `/api/v1/...`，破坏性变更升 v2 |
| 分页 | `page`, `pageSize`, `total`, `items` 统一结构 |
| 错误码 | `code` + `message` + `traceId`；业务错误用 4xx + 明确 code |
| 幂等 | 写操作支持 `Idempotency-Key` header（入职、异动、集成回调） |
| 查询 | 列表支持 `asOfDate` 查询历史组织/任职快照 |

### 3.4 权限模型细化

在 RBAC 基础上，MVP 实现以下数据范围枚举：

```text
SELF | DIRECT_REPORTS | DEPARTMENT | DEPARTMENT_AND_SUB
| LEGAL_ENTITY | ALL（仅系统角色）
```

薪酬相关接口额外叠加 `SALARY_VIEW` / `SALARY_EXPORT` 功能点，与组织数据范围求交集。

### 3.5 技术栈确认（与 README 对齐）

| 层 | 选型 | 备注 |
| --- | --- | --- |
| 前端 | React 19 + Vite + shadcn + React Query | MVP 仅 Admin layout；组件库三端复用，后补 MSS/ESS layout |
| 后端 | Java 17 + Spring Boot 3 + MyBatis-Plus | 复杂查询可用 MyBatis XML |
| 迁移 | Flyway | 每模块独立 migration 前缀：`V1_001__core_org.sql` |
| 缓存 | Redis | 组织树、字典、会话 |
| 文件 | MinIO / 本地 uploads | 合同、证件、导入文件 |
| 搜索 | MVP 用 MySQL LIKE；阶段3 引入 ES | 简历/员工全文检索 |
| 消息 | MVP 用 Spring `@Async` + DB 轮询；阶段2 引入 MQ | 降低早期运维成本 |

## 4. Cursor AI 编码工作流优化

### 4.1 文档分层（供 AI 按需加载）

| 文档 | 用途 | AI 何时读 |
| --- | --- | --- |
| `AGENTS.md` | 项目总览、模块地图、开发顺序 | 每次会话 |
| `docs/前端UI与信息架构规范.md` | Admin 布局、菜单、页面模式、操作体验 | 做前端页面 / Layout 前 |
| `client/src/layouts/AdminLayout.tsx` 等已实现 Admin 页面 | UI 布局与页面模式实践参照 | 新页面开发前对照同模块已实现页面 |
| `HR系统完整系统方案.md` | 业务全景 | 新模块 / 需求澄清 |
| `docs/MVP-AI开发路线图.md` | 可执行任务切片 | 每次开发任务 |
| `docs/领域模型与表设计-MVP.md` | 实体、字段、关系 | 写 entity/API 前 |
| `shared/api.interface.ts` | 接口契约 | 前后端联调 |
| `.cursor/rules/*.mdc` | 分层规范 | 按 glob 自动注入 |

### 4.2 垂直切片开发顺序（严禁水平切）

```text
错误：先写完所有 Entity → 再写所有 API → 再写所有页面
正确：组织模块 Entity → API → 页面 → 联调 → 再员工模块 ...
```

推荐切片顺序见 `docs/MVP-AI开发路线图.md`。

### 4.3 单次 AI 任务粒度

每个 Cursor 任务建议控制在：

- 1–3 张数据库表 + 迁移脚本
- 1 组 REST API（含 DTO + Service + Controller）
- 1 个列表页或 1 个表单页（含 loading/error/empty）
- 或 1 条完整业务流程（如「入职审批通过 → 创建员工」）

### 4.4 Prompt 模板（复制使用）

```text
目标：实现【模块名】的【功能点】
请先阅读：AGENTS.md、docs/领域模型与表设计-MVP.md 中相关章节、shared/api.interface.ts
约束：契约优先；遵循 .cursor/rules；不写 mock；敏感字段脱敏
验收：【列 3–5 条可验证条件】
```

## 5. 风险再评估与缓解

| 风险 | 优化后缓解措施 |
| --- | --- |
| AI 生成代码风格漂移 | AGENTS.md + 分层 rules + 契约文件锁定 |
| 主数据模型返工 | 先冻结 Organization/Employee/Assignment 三实体 2 周 |
| 权限只做菜单 | MVP 验收强制数据范围 + 薪酬字段权限用例 |
| 流程与业务耦合 | Workflow 只存审批链；业务状态机在 Domain 层 |
| 文档过长 AI 漏读 | 拆为 docs/ 下专题文档，AGENTS 只保留索引 |
