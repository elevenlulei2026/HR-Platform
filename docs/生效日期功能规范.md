# 生效日期（Effective Dating）功能规范

> **用途**：为组织、岗位及后续任何需要「历史追溯 / 未来规划 / 版本时间轴」的主数据提供统一设计与开发标准。  
> **何时读**：新增或改造某实体/功能的生效日期能力前，**先读本文**，再对照参考实现（组织、岗位页面）。

---

## 1. 核心概念

**生效日期**是 HR 主数据的时间维度：同一业务编码（如部门编号、岗位编码）可在不同时间点对应不同的有效数据。系统须支持：

| 能力 | 说明 |
| --- | --- |
| **历史追溯** | 查询「截至某日」当时有效的数据快照 |
| **当前维护** | 修正当前生效版本的数据，**不改变**生效日 |
| **未来规划** | 提前录入将来才生效的数据 |
| **版本时间轴** | 同一编码下多条记录，按生效起止日衔接成连续区间 |

这是项目 **架构铁律**（见 `AGENTS.md`）：组织、任职、汇报关系变更必须持久化 `effective_start_date` / `effective_end_date`。

---

## 2. 数据模型约定

### 2.1 标准字段

| 字段（DB） | 字段（API/前端） | 类型 | 含义 |
| --- | --- | --- | --- |
| `effective_start_date` | `effectiveStartDate` | DATE，**必填** | 本条记录开始生效的日期 |
| `effective_end_date` | `effectiveEndDate` | DATE，可空 | 生效结束日；**NULL = 当前仍开放（至今有效）** |

- API JSON 使用 **camelCase**，禁止 snake_case 泄漏到前端类型。
- 日期格式统一：`YYYY-MM-DD`。

### 2.2 快照查询参数 `asOfDate`

| 参数 | 说明 |
| --- | --- |
| `asOfDate` | 格式 `YYYY-MM-DD`；不传则默认**今天** |

**有效判定规则**（列表、树、关联查询通用）：

```text
effective_start_date ≤ asOfDate
AND (effective_end_date IS NULL OR effective_end_date ≥ asOfDate)
```

同一 `code` 若有多条满足条件的记录，取 **`effective_start_date` 最大** 的一条作为该日快照。

### 2.3 多版本唯一约束

版本化主数据（组织、岗位）须将唯一约束从「仅 code」改为：

```text
UNIQUE (code, effective_start_date)
```

同一编码可有多条版本记录，但同一生效日只能有一条。

### 2.4 业务示例

假设今天是 **2026-07-03**，某部门有两条版本：

| 版本 | effective_start_date | 在 2026-07-03 的状态 |
| --- | --- | --- |
| A | 2026-07-01 | **当前有效** |
| B | 2026-07-05 | **将来版本** |

到了 **2026-07-05**：
- A 变为过去数据（`effective_end_date` 自动衔接为 2026-07-04）
- B 变为当前有效数据

## 3. 编辑模式（版本化主数据）

所有支持 **多版本 + 时间轴** 的主数据，编辑分两种模式，通过 `editMode` 区分：

### 3.1 修改当前版本 — `CURRENT`

| 项 | 规则 |
| --- | --- |
| 用途 | 修正名称、负责人、字典字段等，**不改变历史时间轴** |
| 生效日期 | **锁定不可改** |
| 数据操作 | 原地 `UPDATE` 当前版本记录 |
| 典型入口 | 详情 Sheet「编辑」 |

### 3.2 新增生效版本 — `NEW_VERSION`

| 项 | 规则 |
| --- | --- |
| 用途 | 按新的生效日创建一条版本（过去 / 现在 / 将来均可） |
| 生效日期 | **必填**，且须与已有版本的 `effective_start_date` **不重复** |
| 数据操作 | 复制当前版本 + 应用表单修改 → `INSERT` 新版本 |
| 时间轴衔接 | 自动调整相邻版本的 `effective_end_date`，保证区间连续、不重叠 |
| 典型入口 | 详情 Sheet「新增版本」 |

### 3.3 后端校验（组织 / 岗位已统一）

- 新版本须指定生效日期
- 新生效日不能与当前编辑版本相同
- 同一 `code` 下不能已有相同 `effective_start_date` 的版本

### 3.4 时间轴自动衔接逻辑

新增 `NEW_VERSION` 时 Service 层须：

1. 找到新生效日所在区间的**前一版本** → 将其 `effective_end_date` 设为 `新生效日 - 1 天`
2. 找到新生效日之后的**下一版本** → 新版本的 `effective_end_date` 设为 `下一版本生效日 - 1 天`（若无下一版本则为 NULL）

**参考实现**：`OrganizationService.createNewVersion()`、`PositionService.createNewVersion()`。

### 3.5 新建首条记录

- 须填写 `effectiveStartDate`，默认当天
- 首条记录 `effective_end_date` 为 NULL
- 业务编码（部门/岗位）由编码规则自动生成时，新建表单只读提示

## 4. 参考实现：组织管理（部门）

> **路由**：`/admin/org/structure`  
> **页面**：`client/src/pages/admin/org/AdminOrgStructurePage.tsx`  
> **后端**：`OrganizationService.java`、`OrganizationController.java`  
> **契约**：`shared/api.interface.ts` → `Organization`、`OrganizationVersion`、`OrganizationEditMode`

### 4.1 页面结构

| 区域 | 行为 |
| --- | --- |
| **组织树（主区域）** | 全宽展示；树内独立滚动；支持搜索、展开/收起 |
| **顶部工具栏** | **生效日期**日期选择器 + 时态标签（历史快照 / 当前 / 未来预览）；非今天时显示「回到今天」 |
| **树节点悬停** | 快捷操作：添加下级、添加同级、编辑、新增版本、查看详情 |
| **详情 Sheet** | 点击节点后右侧弹出，非常驻分栏 |
| **编辑/新建 Sheet** | 独立抽屉；底部 `SheetFooter` 横向按钮 |

组织因**树形结构**需要较大横向空间，但详情/编辑仍走 **Sheet**，非常驻右侧面板。

### 4.2 详情 Sheet 内容顺序

1. **生效版本时间线**（`VersionTimeline`）— 位于组织名称**上方**
2. 组织名称 + 编码 + 状态
3. 分组字段（基本信息、负责人与 HR 等）
4. 底部操作：**编辑**、**新增版本**、**设为无效**

**版本切换**：点击时间线卡片仅加载该版本详情，**不修改**左侧组织树的 `asOfDate`（避免整树随版本切换跳动）。

### 4.3 编辑 Sheet

- 模式切换：`OptionToggle` —「修改当前版本」/「新增生效版本」
- `CURRENT`：`effectiveStartDate` 字段 **disabled**
- `NEW_VERSION`：`effectiveStartDate` 可编辑且必填；标题为「新增生效版本」
- 保存：`PUT /organizations/{id}`，body 含 `editMode`

### 4.4 API 一览

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/api/v1/organizations/tree?asOfDate=` | 按日期构建组织树 |
| GET | `/api/v1/organizations/{id}` | 单条版本详情 |
| GET | `/api/v1/organizations/by-code/{code}/versions` | 版本时间轴列表 |
| POST | `/api/v1/organizations` | 新建部门（含 `effectiveStartDate`） |
| PUT | `/api/v1/organizations/{id}` | 更新；`editMode`: `CURRENT` \| `NEW_VERSION` |

### 4.5 前端关键符号（便于对照复制）

```text
AdminOrgStructurePage.tsx
├── temporalHint(asOfDate)          // 历史快照 / 当前 / 未来预览
├── VersionTimeline                 // 版本时间线组件（页内定义，可后续抽公共组件）
├── EDIT_MODE_OPTIONS               // CURRENT | NEW_VERSION
├── asOfDate state + 工具栏日期选择
├── openEdit(org)                   // editMode = CURRENT
├── openNewVersion(org)             // editMode = NEW_VERSION
└── handleSave                      // 按 editMode 决定是否提交新 effectiveStartDate
```

### 4.6 数据库

- 组织重设计：`server/src/main/resources/db/migration/V12__organization_department_redesign.sql`
- 字段：`organization.effective_start_date`、`effective_end_date`

| --- | --- | --- |

| 区域 | 行为 |

| **行内操作** | 查看、编辑 |
| **详情 Sheet** | 顶部版本时间线 → 摘要 → 分组字段 → 编辑 / 新增版本 / 设为无效 |

| GET | `/api/v1/positions/by-code/{code}/versions` | 版本时间轴 |
| POST | `/api/v1/positions` | 新建岗位 |
| PUT | `/api/v1/positions/{id}` | 更新；`editMode`: `CURRENT` \| `NEW_VERSION` |

### 5.4 数据库

- 岗位版本化：`server/src/main/resources/db/migration/V14__position_effective_versioning.sql`

## 6. 其他已落地的生效日期场景

以下场景使用 **起止日** 或 **记录有效期**，但**不做**组织那种 `editMode` 多版本复制（除非未来产品明确要求）。

| 场景 | 字段 | 页面/模块 | 说明 |
| --- | --- | --- | --- |
| **任职** | `effectiveStartDate` / `effectiveEndDate` | `AssignmentSection.tsx` | 必填生效开始；空结束=当前任职 |
| **汇报关系** | 同上 + `asOfDate` 查询 | `AdminReportingLinesPage` | 列表支持历史快照 |
| **异动记录** | 异动生效日 | `EmployeeMovementTimeline.tsx` | 按生效日期**倒序**时间线 |
| **证件** | `validFrom` / `validTo` | `archive-field-defs.ts` | UI 标签：生效日期 / 失效日期 |
| **合同** | `effective_date` + 起止日 | 档案子表 | 业务材料字段 |
| **成本中心分摊等子表** | `effective_start_date` / `effective_end_date` | `ArchiveMultiSection` | 多行档案记录有效期 |

**区分**：

- **版本化主数据**（组织、岗位）：同一 `code` 多行 + `editMode` + 版本时间轴 + `asOfDate` 列表/树。
- **记录型有效期**（证件、子表、任职）：通常一行一条时间区间，无 `NEW_VERSION` 复制整条业务对象。

## 6.1 已落地：员工「个人主档」生效日期（版本时间轴）

> **定位**：个人主档属于员工主数据的“版本化主档”，与组织/岗位同级，支持：历史追溯、未来规划、版本时间轴、按快照日期查看。  
> **重要共识**：**永远默认选中“当前有效”（今天）版本**。用户只有在明确操作（时间线点击或日期选择）时才进入历史/未来快照。

### 6.1.1 数据模型

- **版本表**：`employee_master_version`
  - 字段：`effective_start_date` / `effective_end_date`（DATE，`effective_end_date` 为空表示“至今有效”）
  - 唯一约束：`UNIQUE (employee_id, effective_start_date)`（个人主档无业务 `code`，以 `employee_id` 作为版本组）
  - 初始化：为每个 `employee` 回填首条版本（生效开始日默认 `hire_date`）

### 6.1.2 API 约定（对齐组织/岗位）

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/api/v1/employees/{id}?asOfDate=YYYY-MM-DD` | 按日期返回个人主档快照（不传 `asOfDate` 默认今天） |
| GET | `/api/v1/employees/{id}/master-versions` | 返回该员工个人主档版本时间轴（按生效日倒序） |
| PUT | `/api/v1/employees/{id}` | 更新个人主档；body 支持 `editMode` + `effectiveStartDate?`（见下节） |

### 6.1.3 编辑模式（对齐岗位）

个人主档支持两种编辑模式，通过 `editMode` 区分：

- **修改当前版本 — `CURRENT`**
  - 含义：修改当前有效版本的数据
  - UI：生效日期字段 **disabled**
  - 数据：原地更新当前版本记录（不改变生效日）

- **新增生效版本 — `NEW_VERSION`**
  - 含义：指定新 `effectiveStartDate`，基于当前版本复制并创建一条新版本
  - UI：生效日期字段 **必填且可编辑**
  - 校验：新生效日不能与已有版本重复，且不能等于当前版本的生效日
  - 时间轴衔接：自动调整相邻版本 `effective_end_date`，保证区间连续、不重叠（同 §3.4）

### 6.1.4 前端 UI/交互规范（对齐岗位，但位置在「个人主档」区域）

- **显示位置**：生效日期的显示与选择、版本时间线 **必须放在员工详情的「个人主档」区域**（不是页面顶栏）
- **版本时间线 `VersionTimeline`**
  - 放置在「个人主档」卡片内容的最上方
  - 点击某条版本：将快照日期切到该版本的 `effectiveStartDate` 并刷新快照
  - **默认高亮**：与当前 `asOfDate` 命中的版本一致
- **快照日期选择器**
  - 位置：个人主档卡片 toolbar
  - 组件：日期输入 + `temporalHint` 标签（历史快照/当前生效/未来预览）+ 「回到今天」
  - **默认值**：每次打开员工详情时，强制设为今天（当前有效）
- **保存后刷新（必须）**
  - 新增版本或更新成功后：必须立刻刷新版本时间线与快照数据，禁止要求用户关闭重开才能看到新版本

### 6.1.5 参考实现索引

| 层级 | 文件 |
| --- | --- |
| 迁移 | `server/src/main/resources/db/migration/V24__employee_master_effective_versioning.sql` |
| 后端 | `EmployeeService.updateMaster()`、`EmployeeController.listEmployeeMasterVersions()` |
| 前端 | `EmployeeArchiveDetailView.tsx`（个人主档内时间线+日期选择） |

## 7. 新增生效日期能力 — 开发清单

为**新实体**增加与组织/岗位同级的版本化能力时，按以下顺序执行（与 `AGENTS.md` 垂直切片一致）：

### 7.1 设计决策（动手前确认）

| 问题 | 选项 |
| --- | --- |
| 是否同一编码多版本？ | 是 → 走本文「版本化主数据」；否 → 仅起止日字段 |
| 是否需要 `asOfDate` 列表/树快照？ | 通常是 |
| 是否需要版本时间线 UI？ | 有多版本时通常是 |
| 关联数据是否按生效日解析？ | 如上级组织、所属部门须在 `asOfDate` 下解析 |

### 7.2 契约（`shared/api.interface.ts`）

- [ ] 实体类型增加 `effectiveStartDate`、`effectiveEndDate?`
- [ ] `CreateRequest` 必填 `effectiveStartDate`
- [ ] `UpdateRequest` 增加 `editMode?: "CURRENT" | "NEW_VERSION"`、`effectiveStartDate?`
- [ ] 列表/树查询增加 `asOfDate?: string`
- [ ] 增加 `{Entity}Version` 类型（含 `temporal`、`temporalLabel`、`isOpen`）
- [ ] API 类型注释写明路径与参数

### 7.3 数据库（Flyway）

- [ ] 表增加 `effective_start_date`、`effective_end_date`
- [ ] 唯一约束改为 `(code, effective_start_date)`（若有业务编码）
- [ ] 索引：`(code, effective_end_date)` 或等效，便于快照查询

### 7.4 后端（Service 为唯一真相源）

- [ ] 列表/树：`asOfDate` 过滤 + 同 code 取最新 `effective_start_date`
- [ ] `listVersionsByCode(code)` 按 `effective_start_date` 降序
- [ ] `update(id, patch, editMode)`：`CURRENT` 原地更新；`NEW_VERSION` 复制+衔接时间轴
- [ ] 校验：新生效日非空、不重复、不与当前相同
- [ ] 关联校验基于**该版本生效日**的快照（参考 `validateParent`、`resolveParentId`）
- [ ] Controller 鉴权；`editMode` 传入 Service，**禁止**在 Controller 写业务分支

### 7.5 前端

- [ ] API 封装：`getTree` / `list` 传 `asOfDate`；`getVersions`；`update` 传 `editMode`
- [ ] 列表/树顶栏：**生效日期**选择 + `temporalHint` 标签 + 「回到今天」
- [ ] 详情 Sheet：顶部 `VersionTimeline`（参考组织/岗位页内实现）
- [ ] 编辑 Sheet：`OptionToggle` 切换 `CURRENT` / `NEW_VERSION`；`CURRENT` 时禁用生效日
- [ ] loading / error / empty 三态；禁止 mock
- [ ] 遵循 `frontend.mdc`：Sheet、`FormField`、`OptionToggle`、`SheetFooter` 横向按钮

### 7.6 验收

- [ ] 不同 `asOfDate` 返回不同快照
- [ ] `CURRENT` 编辑不改生效日
- [ ] `NEW_VERSION` 创建新版本且时间轴连续
- [ ] 版本时间线可切换查看；行为与组织/岗位一致
- [ ] 同一 code 不可重复相同生效日
